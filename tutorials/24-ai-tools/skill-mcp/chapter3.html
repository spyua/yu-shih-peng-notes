設計判準：我的需求該用哪個？

透過具體判斷標準與場景問答，讓你在實際工作中快速做出正確選擇。

三個判斷框架

面對一個新需求，你只需要回答三個問題，就能快速判斷該用 Skill、MCP，還是兩者並用。

適合做成 Skill
當主要痛點在「方法」，不在「能力」
  • 主要痛點是流程易錯、步驟不穩定、輸出格式不一致，而不是缺乏外部接入能力
  • 已具備 shell、既有腳本、目錄結構與內部樣板，模型只需要更清晰的操作方法
  • 需要將團隊規範、house style、審查標準與風險提醒穩定灌入任務執行
  • 任務主要發生在既有執行環境內，外部整合需求有限
適合做成 MCP
當需要「接入外部世界」，並有治理需求
  • 需要穩定連接外部系統、資料源或企業工具（工單系統、版控平台、知識庫、雲端資源）
  • 需要多個 Agent 或 client 共用同一套能力，且要求清楚的 schema、權限控管與審計紀錄
  • 需要在工具層建立正式邊界，避免模型直接接觸敏感系統或高風險操作
  • 需要跨團隊、跨平台的能力共用與版本穩定性
兩者並用
成熟平台的常態：Skill 編排，MCP 執行
  • 任務既有複雜的流程邏輯，又需要存取外部系統（這是最常見的情況）
  • Skill 負責定義任務流程與操作順序，MCP 提供查詢與操作能力
  • 典型例子：事故分析（Skill 定義分析步驟）+ MCP tools（查 GitHub / K8s / Datadog）

一句話判斷表

把常見的需求陳述拆解成「訊號」，讓你在對話或設計評審中快速定位。

需求描述中出現...建議原因
「AI 老是做錯順序」「輸出格式不穩定」Skill流程方法問題，不是能力問題
「需要接到我們的 Jira / GitHub」MCP外部系統整合，需要協定層
「要讓多個 bot 共用同一套工具」MCP多 client 共用，需要標準接口
「公司規範、審查標準要塞進 AI」Skill知識封裝，不是外部能力
「要在事故分析中自動查日誌」兩者並用Skill 定義流程，MCP 查詢日誌
「需要 schema 穩定、有審計記錄」MCP治理需求，需要協定層
「讓 AI 按我們的報告格式輸出」Skill輸出契約、模板，屬方法層
「還不確定需要接哪些外部系統」先做 Skill投資低，先確認方法論再規劃整合

常見問題 FAQ

❓
Q1：Skill 可以包腳本和模板，是不是「輕量版 MCP」？

這是最常見的誤解之一。腳本和模板是 Skill 的輔助素材，而非其本質。

不是。Skill 的主要角色是封裝程序知識與任務方法；MCP 的主要角色是以標準方式暴露能力。前者偏方法層，後者偏接入層。兩者可以互補，但不宜視為同一物件的輕重版本。

❓
Q2：我們目前只有 Skill、沒有 MCP，架構是否不成熟？

這取決於你目前的場景與需求規模。

不必然。若任務主要在既有執行環境內，且外部整合需求有限，先從 Skill 著手是合理的。然而，一旦需求擴展到多系統存取、跨 client 共用、權限治理與審計，仍需規劃 MCP 或等價整合層。

❓
Q3：可以把所有流程都直接塞進工具，避免再維護一層 Skill 嗎？

聽起來省事，但這是一個常見的架構陷阱。

不建議。這會讓工具過度耦合任務判斷、輸出格式與流程控制，降低可重用性，也會增加測試與維護成本。較健康的做法是讓工具保持能力原子化，再由 Skill 負責任務流程。

❓
Q4：在 coding agent 場景裡，為什麼很多團隊覺得 Skill 已經夠用？

這個觀察本身沒錯，但不能擴大解釋。

因為場景的特殊性。在 shell-first 的 coding agent 場景裡，執行能力往往已由 shell、既有腳本與檔案系統提供，真正欠缺的是穩定的方法論與流程封裝。Skill 的價值會被放大，但這只代表某個場景下的導入優先順序，不代表接入層已消失。

❓
Q5：對外最精確的一句話應該怎麼說？

如果只能用一句話描述兩者關係，建議這樣說：

「Skill 並不取代 MCP；Skill 主要補上程序知識封裝層，而 MCP 仍負責能力接入、標準互通與治理邊界。」

如何精確表述：避免常見的誤導說法

在公開文件、技術演講與社群貼文中，建議避免直接使用「Skill 取代 MCP」這類容易誤導的句型。

❌ 避免說✅ 建議改說
「Skill 取代 MCP」「Skill 讓許多原本不該做成 MCP 的需求，回到更合適的建模方式」
「有 Skill 就不需要 MCP」「Skill 補上了程序知識封裝層，因此團隊不必再把 SOP 塞進工具層」
「MCP 太重了，用 Skill 更好」「MCP 與 Skill 的關係不是替代，而是分工：前者負責能力接入，後者負責任務方法」
三種精確表述

• Skill 讓許多原本不該做成 MCP 的需求，回到更合適的建模方式。

• Skill 補上了程序知識封裝層，因此團隊不必再把 SOP 塞進工具層或 system prompt。

• MCP 與 Skill 的關係不是替代，而是分工：前者負責能力接入，後者負責任務方法。
