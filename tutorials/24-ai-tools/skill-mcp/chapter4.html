企業落地：治理視角與導入順序

從治理角度重新認識分工，以及實務上如何安排導入順序，控制成本與風險。

治理視角：兩者是不同的治理對象

對企業而言，Skill 與 MCP 的差異不僅是設計語意的差異，更是治理對象的差異。把它們混為一談，會導致治理失焦。

🔌 MCP — 平台能力目錄的管理
能力治理
  • 需要當成「平台能力目錄」的一部分來管理
  • 重點：命名規則、schema 穩定性、權限邊界
  • 安全分類：哪些工具有寫入權限？哪些需要雙重確認？
  • 審計機制：每次調用都需要可追查的紀錄
  • 版本管理：schema 變更需要向後相容或版本控制
📋 Skill — 高頻任務的知識產品管理
知識治理
  • 需要被視為「高頻任務的知識產品」來管理
  • 重點：命名與描述的可觸發性（讓 Agent 能找到正確 Skill）
  • 流程版本化：Skill 的更新需要有記錄，支援回滾
  • 輸出契約：定義清楚每個 Skill 的預期輸出格式
  • 回歸驗證：Skill 更新後，需要驗證不破壞既有任務
比喻：企業的兩種資產

MCP 更像企業的設備與系統清冊——需要資安審查、使用授權與定期稽核；Skill 更像企業的標準作業手冊——需要版本管理、定期更新與員工培訓。兩者都是企業資產，但管理方式完全不同。

導入順序：從 Skill 開始，再規劃 MCP

實務上，較合理的導入順序往往是由方法層出發，再向外擴展到整合層。這種順序既能控制成本，也更符合風險治理的節奏。

1

盤點高頻重複任務

找出團隊中最常出錯、最耗時間、輸出最不穩定的任務。這些通常是 Skill 的第一批候選對象。典型例子：技術文件撰寫、程式碼審查、事故報告、週報產出。

2

優先將流程型工作做成 Skill

整理流程、規範、模板與檢查點，製作第一批 Skill。此階段的投資低，見效快，也能讓團隊更清楚哪些任務真正需要外部整合能力。

3

辨識真正需要外部接入的能力

在使用 Skill 的過程中，會逐漸浮現「這個步驟如果能直接查 Jira 就好了」的需求。此時再規劃對應的 MCP server，需求更清晰，設計也更精準。

4

由 Agent 層編排 Skill 與 MCP

最終由 Agent 或 Command 層決策：何時選擇哪個 Skill、何時調用哪個 MCP tool。Skill 定義「怎麼做」，MCP 提供「做得到」的能力，Agent 負責統籌。

為什麼這個順序更好？

先做 Skill 可以讓團隊以低成本驗證方法論，快速看到效果；接著在有足夠需求資訊的基礎上規劃 MCP，可以避免「先建好 MCP server，卻發現根本不知道要接什麼」的浪費。

企業治理對照：管理 Skill vs 管理 MCP

治理面向管理 Skill管理 MCP
命名規則以任務可觸發性為優先，如「製作技術白皮書」以能力語意為優先，如「github.create_pr」
版本管理流程文件版本化，支援回滾Schema 版本管理，向後相容
權限控管控制哪些 Agent 可以掛載哪些 Skill控制哪些操作需要哪個層級的授權
審計需求記錄 Skill 的使用頻率與輸出品質每次工具調用都需要可追查的紀錄
更新流程定期更新流程規範，驗證不破壞既有任務Schema 變更需要通知所有使用方
安全分類哪些 Skill 涉及敏感業務邏輯？哪些工具有寫入/刪除/執行權限？

結論：建模正確，才是真正的問題

「Skill 會取代 MCP」之所以流行，是因為它包裝了一個局部正確、但整體失真的敘事。Skill 的確在許多代理工作流中展現出遠高於傳統 prompt 寫法的穩定度，也降低了部分團隊對協定化整合的即時需求；然而，這並不等於協定層已經失去存在理由。

較嚴謹的結論是：Skill 與 MCP 分別回答不同問題。當一個團隊真正理解「能力接入」與「任務方法」是兩種不同設計工作之後，Skill 與 MCP 的爭論就不再是選邊站，而是建模是否正確、治理是否分層、投資順序是否合理。

問題正確答案
Skill 取代 MCP 了嗎？❌ 沒有。兩者解決不同問題，位於不同層次。
Skill 取代了「什麼」？✅ 取代了把流程塞進工具層、把規範塞進 prompt 的錯誤建模方式。
什麼時候只需要 Skill？任務在既有執行環境內，外部整合需求有限的場景。
什麼時候需要 MCP？需要多系統存取、跨 client 共用、正式治理與審計的場景。
成熟架構應該怎麼做？✅ 兩者並用：Skill 定義流程，MCP 提供能力。

最後一個提醒：成本差異不等於角色等價。做 Skill 的初始門檻較低不代表 MCP 可以省略；能快速見效也不代表適用所有場景。正確的問題永遠是：「這個需求的本質是什麼？應該由哪一層來承擔？」
