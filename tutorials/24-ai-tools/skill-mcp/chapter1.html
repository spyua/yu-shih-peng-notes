核心觀念：Skill 與 MCP 到底是什麼？

用生活比喻帶你搞懂兩者的本質差異，不需要任何 Agent 開發背景。

為什麼這個問題讓人搞混？

自 2025 年以來，「Skill 會取代 MCP」的說法在開發者社群快速流傳。許多人發現，用 Skill 就能讓 AI Agent 做事，不一定要先建 MCP server，感覺 MCP 似乎多餘。

但這個直覺包含了一個推論跳躍：它把「某個場景下不需要先做 MCP」等同於「兩者解同一個問題」。這是理解 Agent 架構最常見的陷阱之一。

生活比喻：廚師技法 vs 食材供應商

Skill 像廚師的食譜與烹飪技法——規定了先醃製、再翻炒、最後擺盤的步驟；MCP 像外部食材供應商的標準訂購系統——負責把新鮮食材安全準時地送進廚房。

廚師再厲害，沒有食材也做不出菜；食材再好，沒有食譜也不知道怎麼烹調。兩者根本不是競爭關係。

讀完本章，你會清楚知道 Skill 和 MCP 各自「是什麼」、「解決什麼問題」，以及為什麼「誰取代誰」是一個錯誤提問。

MCP 是什麼？

MCP 全名 Model Context Protocol，是讓 AI client（如 Claude、GPT）能以標準方式連接外部能力的協定與整合層。依照官方規格，MCP server 可向 client 暴露三類能力：

能力類型用途例子
Resources提供可讀取的資料與上下文公司知識庫、資料庫查詢結果
Prompts提供可參數化的提示模板報告生成模板、審核流程提示
Tools提供可調用的外部操作建立工單、查詢部署狀態、送出 PR
比喻：USB 標準插孔

MCP 就像電腦的 USB 介面規格。不管是滑鼠、鍵盤、隨身碟，只要符合標準，就能插進來使用。MCP 讓任何符合協定的外部服務，都能被 AI client 以同樣方式「接進來」——不必每次重寫接介面。

MCP 的設計核心是「能力接入層」：它關心安全性、權限邊界、schema 穩定性、可互通性，以及如何讓多個 Agent 或 client 共用同一套外部能力。

Skill 是什麼？

Skill 是一種可重用的工作知識包，通常以 SKILL.md 作為入口，並可附帶腳本、模板與參考素材。核心價值不是提供「新的外部能力」，而是告訴模型「這件事應該怎麼做才穩定」。

比喻：新員工訓練手冊

想像你招進一位新工程師，他具備基本程式能力（就像 AI 模型有基本語言能力），但他不知道你們公司的部署流程、程式碼規範與風險提醒。

Skill 就是那本內部訓練手冊——它不賦予員工新的技術能力，而是告訴他「在我們公司，這件事的正確做法是這樣」。

元素說明舉例
SKILL.md主要入口，描述流程與規範「製作 PPT 時，先讀資料再規劃章節」
Scripts可執行的輔助腳本自動產生封面頁的 Python 腳本
Templates輸出格式模板報告的標準 Word 模板
Reference Assets任務中需要的參考素材品牌色彩規範、設計系統文件

核心差異：不是誰比較強，而是誰在不同層

理解這個差異的關鍵，是把「能力是哪來的」和「任務應該怎麼做」這兩個問題分開來看。

MCP
能力接入層 / 整合層
解決「模型能不能連到外部系統」的問題。重視協定、安全、權限、schema 穩定性。
Skill
知識封裝層 / 方法層
解決「這個任務應該怎麼做才穩定」的問題。重視流程一致性、可重用性、輸出品質。
一句話記住差異

MCP 回答：「模型能不能用到外部能力？」

Skill 回答：「模型應該怎麼完成這個任務？」

面向MCPSkill
主要問題能力接入與標準化互通任務方法與知識封裝
核心產出可調用的工具、資源、提示流程文件、模板、腳本
關注重點安全、權限、schema、互通一致性、可重用、任務收斂

「取代說」為何有傳播力？五個根本原因

「Skill 取代 MCP」包含了一個局部正確的觀察：在 coding agent 與 shell-first 工作流裡，執行能力已由 shell 與腳本提供，真正欠缺的是穩定的方法封裝。在這類場景中，Skill 優勢被放大，MCP 需求相對不明顯。

但這只是場景偏差，不是普遍結論。一旦需求擴展到多系統存取、跨 client 共用、正式權限治理與審計，MCP 的不可替代性就會立刻顯現。把特定場景的導入優先順序誤讀為「架構層面的取代」，是本文要釐清的核心誤解。

🔍
原因 1：場景偏差

大量討論集中在 coding agent 與 shell-first 工作流，讓 Skill 的即時效益被放大，卻遮蔽了企業級整合與治理需求。

⚖️
原因 2：比較基準錯置

許多人用「哪個更能讓 AI 做事」比較兩者，但「做事」本身就可能由多層能力共同構成，這個比較式一開始就不嚴謹。

🧱
原因 3：歷史包袱

在缺乏 Skill 容器的時期，許多團隊把 SOP 和決策邏輯硬塞進工具層。Skill 出現後這些內容回到合適位置，卻被誤讀為對 MCP 的替代。

💰
原因 4：成本差異被誤讀

做 Skill 的初始門檻較低，通常只需整理文件與腳本；做 MCP 則涉及協定、安全與平台維運。但成本差異不等於角色等價。

📢
原因 5：傳播失真

「不是每個需求都要先做成 MCP」是合理建議；但在轉述過程中往往被偷換成「有 Skill 就不需要 MCP」。
