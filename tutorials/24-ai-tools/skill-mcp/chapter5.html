圖解範例：看一遍就懂的分工邏輯

用真實場景、流程圖、Before/After 與生活比喻，把 Skill 與 MCP 的分工邏輯刻進直覺。

§1　三個場景・六個比喻

抽象定義看完容易忘記，用你每天會遇到的真實場景映射一次，記憶就會黏上去。

🍳
餐廳廚房
Skill ＝ 食譜 SOP
MCP ＝ 出貨系統接單

食譜告訴廚師「怎麼做這道菜」；接單系統讓廚房連上外送平台收到訂單。兩者缺一不可，角色完全不同。

🏥
醫院
Skill ＝ 診療流程指引
MCP ＝ HIS 醫療資訊系統

「先問主訴、再量血壓、最後開藥」是診療流程；HIS 是讓醫師連到病歷、檢驗結果的標準接入層。

🏗️
建築工地
Skill ＝ 施工規範手冊
MCP ＝ 機具吊掛 API

規範說「澆灌混凝土前要先架好模板」；機具 API 讓你實際操作吊車。知道「怎麼做」和「能做到」是兩件事。

比喻的邊界

所有比喻都只是幫助記憶，不要拉太遠。核心映射就是：Skill = 方法與流程，MCP = 標準化能力接入。

§2　核心問題對比：一眼看出分層

每次拿到一個需求，先問自己這兩個問題。答案自然會把你引到正確的層。

Skill 的問題
「這件事應該怎麼做才對？」

步驟是什麼？先做什麼後做什麼？輸出應該長什麼樣？哪些地方容易出錯需要提醒？這些都是 Skill 要封裝的內容。

MCP 的問題
「模型要怎麼安全地拿到外部能力？」

要連接哪個系統？輸入 schema 是什麼？權限怎麼控管？多個 client 如何共用這個工具？這些都是 MCP 要解決的問題。

再加一題讓你快速定位

「這個需求，主要痛點是流程不穩、還是外部接入困難？」
🔁 流程不穩、容易做錯、輸出不一致
→ 優先考慮 Skill

步驟、規範、範本、判斷條件——這些屬於方法層，用 SKILL.md 封裝最合適。執行能力通常已經存在（shell、腳本、既有工具）。

🔌 需要接到外部系統、多方共用、權限治理
→ 優先考慮 MCP

標準化的 Tools / Resources / Prompts 接入，是 MCP 存在的理由。涉及安全邊界、跨 client 共用、審計紀錄時尤其適合。

※ 同時包含兩者時 → Skill 負責編排，MCP 負責執行外部操作

§3　完整流程圖：事故根因分析

以「Kubernetes 服務異常警報」為例，看一遍 Skill 與 MCP 在真實任務中怎麼分工協作。

🧠
Agent / Command 層
接收警報，判斷意圖

用戶輸入「DB pod 一直 CrashLoopBackOff」，Agent 判斷這是 Incident Triage 任務，載入對應的 Skill。

📋
Skill 層 — 步驟 1
Skill 規定：先確認最近的版本變更

SKILL.md 指定「先查 30 分鐘內的 deployment 記錄，再比對有無 config change」。這個判斷順序是 Skill 封裝的方法知識，不是工具本身。

🔌
MCP 層 — 執行接入
MCP tool：查詢 Kubernetes deployment history

k8s-mcp.get_deployment_events(namespace, since="30m") — 這個能力由 MCP server 提供，有完整的 schema、權限控管與審計紀錄。

📋
Skill 層 — 步驟 2
Skill 規定：比對後若有異動，查 pod log

Skill 的判斷邏輯：「若有新 deployment，優先查對應 pod 的 log；若無，則查 resource quota 與 OOM 事件。」這是 Skill 提供的診斷順序。

🔌
MCP 層 — 執行接入
MCP tool：拉取 pod logs + metrics

k8s-mcp.get_pod_logs(pod_name, tail=200) + prometheus-mcp.query(metric, range) — 兩個不同的 MCP server 提供不同的外部能力。

📋
Skill 層 — 輸出格式
Skill 規定根因報告的格式與風險提醒

「報告需包含：根因假設、受影響範圍、建議行動、升級條件」——這個輸出契約也是 Skill 封裝的內容，與工具層無關。

📤
結果輸出
交付結構化根因分析報告給用戶

最終輸出完全符合 Skill 定義的格式，使用的外部資料由 MCP 提供。兩者在各自的層發揮作用，彼此不越界。

這個範例的關鍵觀察

Skill 貢獻了「查什麼、先後順序、判斷邏輯、輸出格式」；MCP 貢獻了「實際連到 K8s 和 Prometheus 的能力」。把任何一層拿掉，任務都會不完整或不穩定。

§4　三個場景快速分析

每個場景給你「問題是什麼 → Skill 負責什麼 → MCP 負責什麼 → 最終判斷」四格。

🔍
場景 A
PR Code Review 自動化
「讓 AI 幫我 review PR，輸出符合團隊規範的意見」
Skill 負責
先看 diff 大小，超過 500 行先警告
依序檢查：命名、錯誤處理、測試覆蓋
輸出格式：嚴重 / 建議 / 讚美 三類
team house style 規則（從 SKILL.md 載入）
MCP 負責
GitHub MCP：拉取 PR diff 與 file tree
GitHub MCP：讀取 repo 的 .eslintrc / tsconfig
GitHub MCP：留 review comment
（可選）Jira MCP：連結對應 ticket
✅ 結論：Skill 編排 MCP，兩者並用 主要痛點在「review 品質不穩、格式不一致」→ 這是 Skill 的領域；但讀取 PR 和留 comment 需要 GitHub 接入 → 這是 MCP 的領域。
🗄️
場景 B
多個 Agent 共用 Confluence 查詢能力
「文件 Bot、FAQ Bot、Onboarding Bot 都需要查公司知識庫」
Skill 負責
各 Bot 的任務邏輯（各自的 SKILL.md）
搜尋結果的整理與回應格式
找不到資料時的 fallback 流程
MCP 負責
Confluence MCP：統一的搜尋與讀取 tool
統一的 OAuth 權限控管
統一的審計紀錄（誰查了什麼）
schema 穩定，各 Bot 不必各自實作
✅ 結論：MCP 是核心，Skill 是上層 「多個 client 共用同一能力、需要權限控管與審計」→ MCP 的典型使用場景。沒有理由讓三個 Bot 各自爬 Confluence API。
📦
場景 C
Release Checklist 自動執行
「每次發版前，讓 Agent 跑完所有清單，有問題就擋下來」
Skill 負責
清單定義：15 個檢查項目與順序
通過條件與封鎖條件的判斷邏輯
不同環境（staging / prod）的不同規則
輸出：Release Gate 報告格式
MCP 負責
GitHub MCP：確認 PR 都已 merged
CI MCP：確認所有 pipeline 為綠
PagerDuty MCP：確認無 active incident
Jira MCP：確認 release ticket 狀態
✅ 結論：Skill 驅動，MCP 提供多系統能力 清單邏輯與判斷條件是 Skill 的方法知識；真正連到四個不同系統的能力由 MCP 提供。這是最典型的「Skill 編排 MCP」模式。

§5　三種常見錯誤與正確重建

原文白皮書指出 Skill 真正取代的，是三種錯誤建模。用 Before/After 看得最清楚。

錯誤一：把流程邏輯寫死在工具裡

❌ 錯誤做法
MCP tool review_pr() 內部寫死了「先看命名、再看測試、輸出三類意見」
換個團隊規範要改 tool 實作，重新部署
三個不同 Bot 各自 fork 出不同版本的 tool
工具越長越胖，難以測試與維護
✅ 正確做法
MCP tool get_pr_diff() 只負責拉 diff，保持原子化
流程邏輯寫在 Skill 的 SKILL.md，可版本化、可替換
換規範只要更新 Skill，MCP tool 完全不動
MCP tool 可被任何 Skill 重用

錯誤二：把所有 SOP 塞進 system prompt

❌ 錯誤做法
System prompt 長達 8,000 tokens，包含 PR review、release checklist、incident triage 三套 SOP
每次對話都帶入全部規範，浪費 context window
無法針對不同任務按需掛載不同規範
SOP 更新 = 改 prompt = 沒有版本控制
✅ 正確做法
每套 SOP 是獨立的 Skill，有自己的 SKILL.md
Agent 判斷任務類型後，按需載入對應的 Skill
Context window 只在需要時才帶入對應知識
Skill 有版本號，可 A/B test，可回歸驗證

錯誤三：還沒確認整合需求就過早建 MCP

❌ 錯誤做法
任務只在本機 repo + shell 執行，完全不需要連外部系統
但團隊花兩週建了 MCP server，設計 schema、實作 OAuth
真正的問題（流程不穩）一直沒有解決
成本高、見效慢，導致對 MCP 的負面評價
✅ 正確做法
先盤點：任務環境已有 shell + 腳本，主要缺的是方法論
先用一天整理出 SKILL.md，穩定流程
等到真正需要跨系統整合時，再投資 MCP
兩步走：Skill first → MCP when needed
白皮書的精確說法

「Skill 所取代的，是那些把流程誤塞進工具層、把規範誤塞進 prompt、把整合需求與方法需求混為一談的設計。」

§6　快速自我測驗：這個需求該怎麼建？

看每個描述，先在心裡判斷答案，再展開看解析。

需求 Q1
「讓 AI 幫我們統一 Git commit message 的格式與內容規範」

→ Skill。痛點是格式不一致、規範沒有貫徹——這是方法層的問題。commit message 可以由 Agent 在 shell 中產生，不需要外部接入。用 SKILL.md 封裝你的 Conventional Commits 規範即可。

需求 Q2
「讓五個不同部門的 Agent 都能查詢同一個內部 ERP 系統的庫存資料」

→ MCP。五個 client 共用同一能力、需要 OAuth 授權、有審計需求——這是 MCP 的經典場景。不應該讓每個 Agent 各自爬 ERP API。

需求 Q3
「自動化上線前的 smoke test：跑腳本、確認 health check、如有失敗立即通知 Slack」

→ Skill 編排 MCP。「先跑哪個腳本、判斷失敗條件、決定通知層級」是 Skill 的流程知識；「實際執行腳本、打 Slack Webhook」可能需要 MCP（若跨環境或多 client 共用）或直接 shell（若在既有執行環境內）。

需求 Q4
「目前把整套 incident triage SOP 寫在每個 Bot 的 system prompt，想要改善」

→ 重構為 Skill。這正是「把規範誤塞進 prompt」的典型錯誤。把 SOP 提取出來做成 SKILL.md，按需載入，立刻解決 context 浪費與無法版本化的問題。

🎓

你現在已經能用直覺判斷了

流程、方法、規範、SOP → Skill
外部接入、共用能力、schema、治理 → MCP
兩者都需要時 → Skill 編排 MCP
