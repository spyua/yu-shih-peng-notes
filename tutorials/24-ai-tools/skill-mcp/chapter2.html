架構分層：Skill 與 MCP 如何共存？

看懂成熟 Agent 架構的層次結構，以及兩者在真實場景中的分工方式。

企業 Agent 架構的四個層次

要理解 Skill 與 MCP 的分工，最有效的方式是把 Agent 架構畫成「由上到下的分層模型」。每一層各有職責，彼此互補而非競爭。

Layer 1
任務編排層（Agent / Command）

負責決策：「這個任務要用哪個 Skill？要調用哪個 MCP tool？」是整個 Agent 的「指揮中樞」。

Layer 2
知識與方法層（Skill）

負責「怎麼做」：定義流程步驟、規範、模板、檢查點與輸出格式。讓任務執行有穩定方法論。

Layer 3
能力接入層（MCP）

負責「做得到」：以標準協定暴露外部工具、資源與提示。提供查詢系統、操作資料庫、送出請求等能力。

Layer 4
外部世界（API / Services / Data）

真正的外部能力所在：Jira、GitHub、資料庫、雲端服務、知識庫。MCP 負責把這些標準化地接進來。

關鍵判讀

Skill 定義「怎麼做」，MCP 提供「做得到」的標準接入；真正的外部能力仍位於 API、服務與資料層。

Skill 是醫學診療規範（問診 → 理學檢查 → 抽血 → 影像 → 診斷的標準程序）；MCP 是各種醫療設備的操作介面（X 光機、血液分析儀、MRI 的標準操作協定）。

診療規範告訴醫生「該做什麼檢查、按什麼順序」，醫療設備提供「做得到這些檢查的能力」。兩者缺一不可。

Skill 真正取代了什麼？

若必須精確描述這波變化，較合理的說法是：Skill 正在取代某些錯誤的能力建模方式，而不是從根本上取代 MCP。

✗

被取代的錯誤一：把流程與判斷邏輯寫死在工具層

過去許多團隊把「先做 A 再做 B」的流程控制塞進 MCP tool 裡，導致工具過度耦合業務邏輯，難以重用與測試。Skill 讓流程回到更合適的方法層。

✗

被取代的錯誤二：把大量任務規範塞入 system prompt

把所有 SOP 和規範塞進 system prompt，導致內容無法版本化、難以測試、也難以針對特定場景按需掛載。Skill 讓這些知識有了合適的容器。

✗

被取代的錯誤三：過早投資重量級協定化能力

在尚未確認整合需求前，就為所有任務建立 MCP server，增加了不必要的工程成本。Skill 讓低整合需求的任務有了輕量替代方案。

精確的結論

Skill 的價值是讓能力分層變得更乾淨：該在整合層處理的，留在整合層（MCP）；該在知識與方法層處理的，回到知識與方法層（Skill）。

常見混淆點：有人認為既然 Skill 可以包含腳本與模板，本質上就是「輕量版 MCP」。這是不對的。Skill 的主要角色是封裝程序知識；MCP 的主要角色是以標準方式暴露能力。前者偏方法層，後者偏接入層——不宜視為同一物件的輕重版本。
