操作策略與實戰情境

用三個真實場景，看懂理解與不理解這些概念的人，在使用 Agent Tool 時的具體差異。

📖 你會學到什麼
  • 面對多檔案重構任務時，如何用拆分策略避免 Context 溢出
  • 為什麼對簡單任務用旗艦模型反而會「壞事」，以及正確的模型選擇邏輯
  • Debug 複雜問題時，精準投餵資訊比大量丟給 AI 更有效的原因
  • 從「被動使用者」到「主動操控者」的核心心態轉變

情境一：重構跨 10 個檔案的模組

這個場景展示了 Context 管理能力 的差距。面對一個涉及多個檔案的重構任務，兩種人的做法和結果截然不同。

❌ 不理解的做法
直接在一個 Session 裡對 Agent 說「幫我重構這個模組」。Agent 開始讀取所有相關檔案，到第 7 個檔案時 Context 已佔滿 80%。Agent 開始「忘記」前面讀過的內容，產出的程式碼與前幾個檔案的介面不一致，甚至重複定義了已存在的函式。
結果：程式碼不一致，使用者只覺得「AI 好笨」
✓ 理解的做法
先規劃拆分策略——把 10 個檔案依相依性分成 3 組。每組用一個乾淨的 Session 處理，完成後 /clear 再開下一組。每組開始前，只餵入該組需要的檔案和前幾組產出的介面定義（而非全部原始碼）。
結果：每個 Session 的 Context 使用率都在 50% 以下，產出一致且正確

關鍵策略：依相依性拆分任務

1

分析檔案相依性

在開始之前，先找出哪些檔案彼此緊密相依，把它們分成 2–3 組。

2

每組一個 Session

用 /clear 開始乾淨的 Session，只讀取當前組的檔案，避免 Context 被其他組的資訊佔用。

3

傳遞介面定義，而非全部原始碼

下一組的 Session 只需要「前幾組產出的介面摘要」，不需要完整的原始碼，這樣 Context 消耗極少。

差異關鍵： 理解 Context Window 有限性 → 主動拆分任務，避免 Context 溢出導致品質下降。

情境二：用 Agent 寫一個簡單的 CRUD API

這個場景展示了 模型匹配能力 的差距。同樣的任務，選錯模型不只浪費錢，還會讓 AI「想太多」。

❌ 不理解的做法
開著旗艦模型（Opus），把 Reasoning Effort 設到最高，然後讓 Agent 生成一個標準的 CRUD endpoint。等了 45 秒才拿到結果，而且 Agent「想太多」——自動加了分頁、軟刪除、審計日誌等根本沒要求的功能。高 Reasoning 還把 Context 空間吃掉一大塊。
結果：45 秒、功能過多、Context 快速消耗
✓ 理解的做法
判斷這是低複雜度任務，切換到標準模型（Sonnet），Reasoning Effort 設為 low。3 秒內拿到乾淨的 CRUD 程式碼，沒有多餘功能。Context 消耗極少，同一個 Session 還能繼續處理後續任務。
結果：3 秒、精準符合需求、Context 消耗極少

模型選擇決策樹

任務類型建議模型建議 Reasoning
簡單 CRUD、格式轉換、快速問答 Haiku / Mini minimal / low
中等複雜功能、一般 bug fix Sonnet / Standard low / medium
架構設計、跨模組重構 Opus / Max medium / high
深度 debug、複雜依賴分析 Opus / Max high / xhigh

差異關鍵： 理解模型能力等級與 Reasoning 消耗 → 根據任務複雜度選擇合適的模型和思考強度，避免「殺雞用牛刀」。

情境三：Debug 跨服務的生產問題

這個場景展示了 資訊篩選能力 的差距。把所有資訊丟給 AI，和精準投餵關鍵資訊，結果天差地遠。

❌ 不理解的做法
把所有 log、設定檔、原始碼一股腦貼進去，希望 AI「自己看出問題」。Context 瞬間被灌滿，Agent 只能看到最後貼進去的片段，完全忽略了最前面貼的關鍵設定檔。反覆在同一 Session 補充資訊，每次都讓情況更糟。
結果：結論完全偏離方向
✓ 理解的做法
先自己縮小範圍——從 log 中提取關鍵的錯誤訊息和時間軸（而非整份 log），只提供相關服務的設定檔片段。使用旗艦模型搭配高 Reasoning Effort。精準提供上下文，讓 Context 主要用在「思考」而非「儲存無關資訊」。
結果：精準定位 Service B 的 timeout 設定與 Service A 的 retry 策略衝突
精準投餵的原則

不是「給 AI 越多資料越好」，而是「給 AI 最相關的資料」。把有限的 Context 空間留給「思考」，而不是「儲存無關資訊」。你的任務是在把問題丟給 AI 之前，先做一次初步的篩選。

差異關鍵： 理解 Context 的三層結構 → 精準控制「餵進去的資訊量」，把有限的 Context 留給真正重要的內容。

從被動使用者到主動操控者

理解這些基礎概念，本質上是從「被動使用者」轉變為「主動操控者」——你不再是把任務丟給 AI 然後祈禱，而是有策略地分配 AI 的有限資源。

能力維度不理解的人理解的人
Context 管理 被動等 AI 出問題 主動規劃，預防溢出
模型選擇 永遠用最貴的 依任務匹配，省時省錢
資訊提供 全部丟給 AI 篩選精華，精準投餵
問題歸因 「AI 好笨」 「我的操作策略需要調整」
核心心態轉變

有了這些理解，你不再是被動地使用 AI——你是在有策略地分配 AI 的有限資源，讓每一個 token 都花在刀口上。把 AI 當成一個有限資源來管理，而不是一個無限能力的神器。

❓ 問答練習
重構跨 10 個檔案的模組時，為什麼不能一次全部丟給 Agent？ +
因為每讀一個檔案都在消耗 Context。10 個檔案可能超過 30,000–50,000 tokens，佔掉 Context 的 15–25%，加上系統指令和對話歷史，Context 很快就會接近上限。AI 開始「忘記」先讀的檔案，導致後來產出的程式碼與前面的介面不一致，甚至重複定義已有的函式。分批處理、每批一個 Session，才能確保每次 AI 都在完整的工作記憶下運作。
在多 Session 的重構任務中，要傳遞什麼給下一個 Session？ +
傳遞介面定義與摘要，而不是完整原始碼。具體來說就是：函式簽名（名稱、參數、回傳型別）、模組的 export 清單、以及前一批次的關鍵架構決策。這讓下一個 Session 能夠保持一致性，同時只消耗少量 Context——而不是把前幾個 Session 讀過的所有原始碼再全部塞一遍。
Debug 複雜問題時，為什麼「全部丟給 AI」反而更難得到正確答案？ +
有兩個原因。第一，Context 被無關資訊佔滿：大量 log 和設定檔湧入，讓 AI 難以在其中識別真正關鍵的訊號，反而容易被雜訊帶偏。第二，AI 的注意力機制有距離偏差：離最後輸入越遠的內容，AI 越容易忽略——如果你最先貼的關鍵設定檔被後來的大量 log 淹沒，AI 很可能就漏看了。先自己做初步篩選，才能讓 AI 把「腦力」用在分析而不是篩選。
「主動操控者」和「被動使用者」在日常操作上的具體差異是什麼？ +
被動使用者把任務丟給 AI 然後等待，出問題就怪 AI 笨。主動操控者在下指令之前會先思考：這個任務適合哪個模型等級？需要多高的 Reasoning？要給 AI 哪些資訊、哪些可以省略？這個 Session 的 Context 還夠嗎？需不需要先 /clear？ 這些判斷都在任務開始前就做好，而不是被動等問題浮現。核心差異是：把 AI 的有限資源當成需要主動分配的東西，而不是假設它無限充足。
