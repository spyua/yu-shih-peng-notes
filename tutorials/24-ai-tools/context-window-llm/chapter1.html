LLM 模型基礎

搞懂大型語言模型的本質、能力等級與 Reasoning 機制，為操作 Agent Tool 打下基礎。

📖 你會學到什麼
  • LLM 的本質是什麼，為什麼它能處理的不只是「接龍」
  • 模型能力等級（小型 / 標準 / 旗艦）的實際差距，以及如何根據任務選擇
  • Reasoning 是什麼，以及它對任務品質的影響
  • Token 的概念與快速估算方式，為後續 Context 管理打基礎

什麼是 LLM？

LLM（Large Language Model，大型語言模型）是一個經過海量文本訓練的語言模型。它的核心機制說起來其實很簡單——

生活比喻：超強文字接龍選手

你可以把 LLM 想成一個極度博學的文字接龍高手：你給它前文，它就接後文。它讀過的文字量，比任何一個人一輩子能讀完的都多出幾個數量級。

但 LLM 不只是接龍。在大規模訓練之後，它展現出了推理、程式碼生成、多語言理解等「訓練時沒有明確教過」的能力——這被稱為 湧現能力（Emergent Abilities）。這讓它從「文字預測器」升級為「通用任務執行器」。

模型的能力等級

不同的 LLM 有不同的能力等級。選擇哪個等級的模型，直接影響任務品質、速度和成本。

維度 小型（Mini/Light） 標準（Standard） 旗艦（Max/Reasoning）
推理深度 淺層推理，適合直覺型任務 中等推理，能處理多步驟邏輯 深度推理，能處理複雜架構決策
上下文理解 有限，容易忽略遠處的資訊 良好，能整合中等長度的上下文 優秀，能在大量資訊中找到關聯
速度 快 中等 慢
成本 低 中等 高

主流模型對應

小型
Haiku / GPT mini
速度快、成本低。適合簡單 CRUD、格式轉換、快速問答。
速度快 成本低
標準
Sonnet / GPT standard
日常開發主力。能處理中等複雜任務，兼顧速度與品質。
均衡 主力
旗艦
Opus / GPT max
複雜任務首選。架構設計、跨檔案重構、深度 debug 的最佳選擇。
最強 最貴

常見迷思：不是越強越好。 永遠用旗艦模型是錯的——旗艦模型速度慢、成本可能是小型的 10–50 倍，而且太強的模型有時會「想太多」，對簡單任務產出冗餘結果。正確做法是根據任務複雜度匹配模型。

Reasoning：讓模型先想再答

一般的 LLM 回答是「直覺式」的：看到問題就直接產出答案。但 Reasoning 模式讓模型在回答之前，先進行一段內部的思考推理過程。

生活比喻：打草稿 vs. 直接交卷

沒有 Reasoning，就像學生看到數學題直接把答案寫上去。有 Reasoning，就像先在草稿紙上把解題步驟寫一遍，確認無誤再填答案。複雜題目的差異，是天壤之別。

Reasoning 的實際影響

場景 無 Reasoning 有 Reasoning
簡單 CRUD 兩者差異不大 略慢，但結果相同
多檔案重構 容易遺漏邊界情況 會主動考慮影響範圍
架構決策 給出「看起來對」的方案 會權衡多種方案的 trade-off
Debug 複雜問題 表面修復 追蹤根因

Reasoning Effort 等級（以 Codex 為例）

不同的 Agent Tool 提供不同程度的 Reasoning 控制。以下是 Codex 的 model_reasoning_effort 設定：

minimal
幾乎不思考，最快
low
輕度思考
medium
中等思考（預設）
high
深度思考
xhigh
極深度，最慢最強

Reasoning 也消耗 Context。 思考過程產生的 thinking tokens 會佔用 Context 空間。Reasoning Effort 開太高，除了速度變慢之外，也會加速 Context 的消耗——這在長任務中是需要主動管理的風險。

Token：AI 的計量單位

Token 是 LLM 處理文字的最小單位。它不是字元，也不是單字，而是模型的 tokenizer 切分後的片段。

生活比喻：積木塊

把文字想像成一堆積木。Token 就是那些積木塊——有些字是一塊積木，有些詞是一塊，有些符號也是一塊。整個 Context Window 能裝多少積木，就是它的大小上限。

Tokenization 範例 
# 英文
"Hello world"  →  ["Hello", " world"]  = 2 tokens

# 中文
"你好世界"      →  ["你好", "世界"]    = 2 tokens

# 程式碼（切分更細）
"function getData() {"  →  5 tokens

快速估算指南

內容類型每 1,000 tokens 約等於
英文文本750 個單字
中文文本500–700 個字
程式碼100–150 行
為什麼要理解 Token

Context Window 的單位就是 Token（Claude 4 系列 = 200K tokens）。API 計費的單位也是 Token。理解這個單位，才能有效估算「讀取這個檔案會消耗多少 Context」。一個 500 行的程式碼檔案大約 3,000–5,000 tokens。

❓ 問答練習
LLM 的「湧現能力」是什麼意思？它和一般的訓練有何不同？ +
湧現能力（Emergent Abilities）指的是模型在訓練過程中從未被明確教過、卻在規模足夠大之後自然出現的能力，例如推理、程式碼生成、跨語言理解。這不像傳統 ML 任務那樣是一對一的訓練目標，而是在大量語言學習後「意外」獲得的能力——規模到了，能力就突破訓練邊界。
為什麼對簡單任務使用旗艦模型反而可能是壞事？ +
有三個原因：速度慢（旗艦模型回應時間長，簡單任務用它是浪費時間）、成本高（API 場景下費用可能是小型模型的 10–50 倍）、過度思考（太強的模型會對簡單問題「想太多」，反而加入不必要的功能或邏輯，讓結果變複雜）。正確做法是依任務複雜度匹配模型。
Reasoning Effort 設得越高越好嗎？什麼情況應該調低？ +
不是。Reasoning Effort 越高，思考時間越長、消耗 Context 也越多。適合調高的場景是：架構決策、跨模組重構、複雜 Debug 等需要深度推理的任務。應該調低的場景是：格式轉換、簡單 CRUD、快速問答等直覺型任務。開太高的 Reasoning 不只慢，還會把寶貴的 Context 空間吃掉，影響後續任務。
一個 500 行的 Python 檔案大約消耗多少 tokens？ +
大約 3,000–5,000 tokens。程式碼的 tokenization 比純文字細——每 1,000 tokens 約等於 100–150 行程式碼。所以 Agent 讀取一個 500 行的檔案，可能就消耗掉 Claude 4 系列 200K 總 Context 的 1.5–2.5%。如果一個任務要讀 10 個類似的檔案，光讀檔就可能消耗掉 15–25% 的 Context。
